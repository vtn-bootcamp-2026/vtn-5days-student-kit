# UDDS Codex Slide Generator

This is a local Codex skill derived from `udds-slide-generator`, but the slide image redesign layer composes the **Codex `$imagegen` system skill using built-in `image_gen` mode** instead of a Python script calling `gemini-3.1-flash-image-preview` or similar image-preview APIs.

Use this skill for two production cases, then assemble the resulting 16:9 slide images into a PowerPoint deck:

- **Case A: Redesign From Old PDF.** Convert an existing PDF/deck into page images, classify each page, extract/refine content, and redesign each slide into a selected theme and moodboard.
- **Case B: Create Design From Document.** Read a source document/article/brief, create a new slide plan and `deck_manifest.json`, then generate slides from content plus brand skeletons without old slide page images.

## Output Root Rule

For production deck work, keep all deck artifacts under a single `outputs/` project folder. Do not split one deck across `reports/` and `outputs/`.

Recommended structure:

- `outputs/slides/<project>/<version>-work/`: working source pages, `deck_manifest.json`, `archetype_review.md`, image jobs, QA jobs, QA reports, recorder metadata, and generated slide images.
- `outputs/slides/<project>/<version>-images/`: copied final slide image set used as the PPTX assembly source of truth, when a separate deliverable image folder is useful.
- `outputs/slides/<project>/<version>.pptx`: final image-based PPTX deliverable.

Use `reports/` only for non-slide operational logs when the user explicitly asks for a report outside the deliverable tree. For slide production, QA and verification files belong beside the deck under `outputs/`.

## Important Boundary

- **Codex-only skill:** This workflow requires Codex's `$imagegen` system skill and built-in `image_gen` tool. Antigravity and Gemini coding assistants cannot call Codex built-in `image_gen`, so they must not use this skill to create slide images.
- **Hard rule:** Final slide images MUST be created with Codex's built-in `image_gen` tool only. Do not synthesize, render, mock, or "good enough" final slide images with Python scripts, Pillow/PIL, ImageMagick, canvas/SVG renderers, browser screenshots, local templates, or any other non-Codex image-generation path.
- Scripts in this skill are support utilities only. They may extract source pages, prepare prompts/jobs, validate inputs, rename files for fix mode, assemble PPTX files, and record/copy the selected Codex-generated image. They must not create the final redesigned slide artwork.
- Do **not** run the legacy Gemini image-preview scripts for slide rendering.
- Do **not** call Gemini image generation APIs directly from Python for Stage 3.
- Do **not** write or run a replacement Python client for the OpenAI image API unless the user explicitly asks for `$imagegen` CLI/API fallback.
- Do **not** use Pillow/PIL to compose a slide, draw text, paste logos, resize layers into a finished slide, or generate placeholder final images.
- Do **not** treat `prepare_image_tool_jobs.py` as an image generator. It creates prompt files and `image_tool_jobs.json`; it does not fulfill any job.
- Do **not** assume built-in `image_gen` can write directly to `output_path`.
- Use `scripts/prepare_image_tool_jobs.py` to create prompt files and `image_tool_jobs.json`.
- **Optimization (GPT-Image-2):** Prompts generated by this script follow the **Sequential Weighting Protocol**, putting critical info (16:9, Archetype) in the first 15 words. They also include `(don't change the prompt, send it as it is.)` to suppress unwanted AI "enhancements".
- For each prepared job, load the listed local input images with `view_image`, invoke Codex's built-in `image_gen` once through the `$imagegen` system skill, then record the selected `$CODEX_HOME/generated_images/...` result with `scripts/record_image_tool_result.py`.
- The generated slide images must end up at the `output_path` listed in `image_tool_jobs.json`, because `assemble_pptx.py` expects those filenames.

Follow the Codex system `$imagegen` skill exactly for the actual native image invocation policy:

- Default path is built-in `image_gen`.
- Built-in `image_gen` does not require `OPENAI_API_KEY`.
- Built-in generated files are saved under `$CODEX_HOME/generated_images/...` by default.
- Project-bound outputs must be copied or moved into the workspace.
- CLI/API fallback is only allowed if the user explicitly asks for it or explicitly confirms a fallback that `$imagegen` says requires confirmation.
- For many slides, call built-in `image_gen` once per slide job rather than trying to batch through a local script.
- If Codex's built-in `image_gen` tool is unavailable, stop and ask the user how to proceed. Do not silently fall back to Pillow, local rendering, Gemini, or direct OpenAI API calls.
- If multiple Codex chats work on the same deck in parallel, split ownership by explicit job ranges (for example `slide1`-`slide10`, `slide11`-`slide20`, `slide21`-`slide30`). Each chat must record each image immediately after its own `image_gen` call. Do not let multiple chats work on the same `job-id`.

## Advanced Prompting Techniques (GPT-Image-2)

When invoking built-in `image_gen` through `$imagegen`, the prompt must adhere to these deterministic design principles:

1. **Sequential Weighting:** The first 15 words decide 80% of the output. Always lead with `[16:9 Aspect Ratio]. Create one finished [Archetype] presentation slide.`
2. **Prompt Suppression:** Always include `(don't change the prompt, send it as it is.)` to prevent the model from deviating from your technical spec.
3. **Text Fidelity:** Use keywords like `VERBATIM`, `RENDER EXACT TEXT`, and double quotes for all Vietnamese content. Explicitly forbid translation or paraphrasing.
4. **Spatial Anchoring:** Use specific physical terms like `Top-right logo block`, `Center-aligned title`, `Left 50% split image` to guide the thinking mode's layout planning.
5. **Change + Preserve + Physics (for Tweaks):** When polishing a slide, define what to **Change** (the issue), what to **Preserve** (text, colors, layout), and the **Physics** (lighting, shadows) to ensure a seamless result.

## Visual Authority Rule

UDDS layout archetypes (`LT-*`) and core skeleton images are **layout geometry references only**.

- Use LT skeletons for spatial structure: zones, anchors, hierarchy, flow direction, safe areas, and logo position.
- Do **not** copy colors, backgrounds, textures, illustration style, effects, or accent treatments from core LT skeletons.
- The selected theme and moodboard are the only visual-style authority: palette, background, typography mood, illustration style, texture, lighting, and decorative motifs.
- If a core skeleton conflicts with the selected moodboard, follow the moodboard and keep only the skeleton geometry.

## Workflow Overview

### Case A: Redesign From Old PDF

Use this path when the user has an old PDF/deck and wants it redesigned into a theme/moodboard.

1. Convert PDF pages to PNG with `extract_pdf.py`.
2. Prepare and execute OpenAI-native visual triage with `prepare_triage_jobs.py`.
3. Write and review `archetype_review.md`.
4. After approval, write `deck_manifest.json`.
5. Refine content density and `imagegen` instructions.
6. Prepare image jobs with `prepare_image_tool_jobs.py`.
7. Run `$imagegen` in built-in `image_gen` mode once per slide job and record outputs.
8. Assemble the image-based PPTX draft.
9. Run Gemini image QA as the final decision/reporting step.

For this case, `deck_manifest.json` should usually use:

- `imagegen.intent`: `edit`
- `imagegen.input_images`: source page image as Image 1, brand skeleton as Image 2

### Case B: Create Design From Document

Use this path when the user has a source document, article, outline, or brief and wants a new deck designed from scratch.

For an OpenAI-native run, the Codex agent reads the source text directly and authors `deck_manifest.json` plus `archetype_review.md`. There is no Gemini text-planning script in this skill.

Prepare a planning scaffold inside the deck's output work folder:

```powershell

  --input <source-document.md> `
  --output outputs/slides/<project>/<version>-work
```

Then read `openai_document_deck_prompt.md`, author `archetype_review.md` and `deck_manifest.json`, and show `archetype_review.md` to the user for approval/corrections before image generation.

Output:

- `deck_manifest.json`
- `archetype_review.md`
- `openai_document_deck_jobs.json`
- `openai_document_deck_prompt.md`

For this case, `deck_manifest.json` should usually use:

- `imagegen.intent`: `generate using reference images`
- `imagegen.input_images`: LT skeleton as the layout geometry reference; theme/moodboard as the visual-style authority
- `page`: a synthetic stable page name such as `page_1.png` even when no source page image exists

Then continue to content refinement and image job preparation. `prepare_image_tool_jobs.py` automatically detects missing source page images and prepares Codex built-in `image_gen` tool jobs from manifest content plus skeleton. It does not generate or render slide images itself.

### Targeted Slide Fix Mode

Use this mode only for localized fixes to an already generated slide:

- Vietnamese typo or diacritic error
- Wrong accent/color drift
- Minor visual artifact

Switch to this mode automatically when the user names one or more already-generated slides and asks for a localized fix, such as "slide 5 sai typo" or "slide 9 chỉnh màu". Do not regenerate the whole batch for these cases.

Do not use targeted fix mode for layout errors, wrong archetype, bad structure, or major content density problems. For those cases, remove or move aside the generated slide and regenerate that slide from the source page/document plus LT geometry.

Targeted fix mode uses only the current generated slide as the edit target. It does not attach the LT skeleton, logo reference, source page, theme images, or moodboard images. The prompt must preserve the slide exactly except for the requested fix.

For typo/OCR/diacritic repairs on already rendered slide images, use Codex's built-in `image_gen` edit path by default. Do not use deterministic bbox text overlay as the default typo path. The standard typo prompt should be short and preservation-heavy:

> Dùng Image A làm ảnh gốc. Giữ nguyên toàn bộ bố cục, màu sắc, font, icon, hình minh họa và nội dung slide. Chỉ sửa lỗi typo trong phần chữ, đặc biệt lỗi sai dấu tiếng Việt, thiếu dấu, nhiễu dấu, nhầm ký tự hoặc chữ bị méo. Render lại chữ rõ ràng, sắc nét, đúng chính tả. Không tự ý thay đổi nội dung khác, không thiết kế lại slide. Tạo lại `<exact corrected text>` cho `<target area>`.

Rules for typo fixes with `image_gen`:

- Back up `slide<TOKEN>_redesigned.jpg` to `slide<TOKEN>_redesigned_to-be_fixed.jpg` before patching.
- Use only the current slide image as Image A / Image 1. Do not attach skeletons, moodboards, logos, or source pages for typo fixes.
- Keep the prompt simple: state the exact corrected text and the target area; do not over-specify layout or redesign instructions.
- Ask `image_gen` to preserve layout, colors, font, icons, illustrations, and all unrelated slide content.
- Explicitly mention Vietnamese typo classes: sai dấu, thiếu dấu, nhiễu dấu, nhầm ký tự, chữ bị méo.
- After generation, inspect the result and record the selected Codex-generated image into `slide<TOKEN>_redesigned.jpg`.

Vietnamese typo prompt guidance:

- Match the existing typography exactly: font family category, weight, size, style, color, casing, perspective, texture, shadow, and baseline alignment.
- Inpaint corrected text so it looks native to the slide, not pasted on top: preserve local background, opacity, gradient/material treatment, antialiasing, edge sharpness, text box boundaries, line height, letter spacing, kerning rhythm, wrapping, surrounding whitespace, and nearby layout anchors.
- Do not switch between serif and sans-serif, do not force ALL CAPS unless the surrounding text already uses ALL CAPS, and do not rewrite Vietnamese sentence case into Initial Caps.
- Quote the exact corrected text and ask for `VERBATIM` rendering. Explicitly forbid translation, paraphrasing, extra words, pseudo-text, and changes to unrelated text.
- For difficult Vietnamese marks, describe the accent geometry in English when useful: `circumflex accent`, `breve accent`, `horn accent`, `acute accent`, `grave accent`, `hook accent`, `tilde accent`, and `dot under the letter`.
- For stacked marks such as `ầ`, `ề`, `ở`, `ưỡ`, or `ậ`, state that the base vowel, vowel modifier, and tone mark must stay visually distinct with a small gap so accents do not merge.
- Prefer a minimal edit prompt over a full slide prompt. Example: `Use Image A as the source slide. Preserve the entire layout, colors, icons, illustration style, font style, shadows, and all unrelated text. Replace only the misspelled title text with VERBATIM "quyền". The letter "ê" must have a circumflex accent with the grave accent stacked above it, both clean and distinct. Do not redesign the slide.`
- If `image_gen` fails to render the corrected Vietnamese text after 1-2 careful attempts, stop and ask the user before simplifying text. Do not silently convert to unaccented Vietnamese, English, icons, or deterministic bbox overlay.

Deterministic bbox text overlay is a fallback only when the user explicitly asks for it or when `image_gen` repeatedly fails and the user approves the fallback. Do not silently switch typo fixes to local raster text overlay.

Standalone typo skill decision:

- Keep UDDS slide typo repair inside this skill when the asset is an already generated UDDS slide, because this skill owns the Codex-only `image_gen` boundary, backup naming, job manifest, recorder, QA, and PPTX assembly contract.
- Create a separate typo-repair skill only if the workflow must support non-UDDS images, non-slide assets, or cross-project typo repair without the UDDS manifest/output conventions.

For multiple targeted fixes in the same batch, process them sequentially and record each one before preparing the next fix. Example: fix `slide5`, run `$imagegen`, record `slide5_targeted_fix`, QA slide 5, then fix `slide9`. Do not prepare several targeted fix jobs into one shared manifest unless the user explicitly asks for a custom batch-fix manifest.

### Stage 4. Run Codex Built-in `image_gen` Tool For Each Job

For each pending job in `image_tool_jobs.json`:

1. Read the job's `prompt_path`.
2. For every listed `input_images[].path`, call `view_image` first so local files are visible to built-in image edit/generation.
3. Invoke Codex's built-in `image_gen` tool using the prompt exactly as the visual spec. This is the only allowed step that creates the final slide artwork.
4. Inspect the generated output privately for text fidelity, 16:9 composition, brand consistency, and artifact removal.
5. Select the best generated image from `$CODEX_HOME/generated_images/...`.
6. Record it into the expected slide output path:

```powershell
python ".agents/skills/udds-codex-slide-generator/scripts/record_image_tool_result.py" `
  --jobs outputs/slides/<project>/<version>-work/redesigned/image_tool_jobs.json `
  --job-id slide1 `
  --source <selected-image-tool-output.png>
```

Repeat until every job is `complete`.

Do not leave final slide assets only in `$CODEX_HOME/generated_images/...`. Always run `record_image_tool_result.py` or otherwise copy the selected asset to the job's `output_path`.

Never mark a job complete using an image that came from Pillow/PIL, local compositing scripts, browser screenshots, SVG-to-PNG rendering, or a placeholder generator. A completed job must point to an image created by Codex's built-in `image_gen` tool and then recorded into the expected workspace path.

Parallel batch rule: if three chats generate the same deck at the same time, they may share one output folder only when each chat owns a disjoint set of `job-id`s and records immediately per job. The recorder refuses to overwrite a job already marked `complete`; an intentional replacement requires `--allow-overwrite-complete`. For maximum safety, use separate output folders/manifests per batch and merge only after QA.

Quality requirements for Codex built-in `image_gen` tool prompts:

- The output must be 16:9.
- Preserve Vietnamese text and diacritics exactly.
- Use the skeleton for layout geometry only: zones, anchors, hierarchy, flow direction, safe areas, and logo position.
- Use the selected theme and moodboard for visual DNA: palette, background, typography mood, illustration style, texture, lighting, and decorative motifs.
- Never copy color accents, backgrounds, or decorative treatments from a core LT skeleton when they conflict with the selected moodboard.
- Sanitize third-party logos, third-party visual marks, and dummy placeholder text.
- Do not add explanatory captions or process text.
- Treat slide jobs as `productivity-visual`.
- Use the skeleton as a supporting geometry reference; use the source/current slide as the edit/content target when present.

### Stage 5. Assemble PPTX

After all redesigned slide images are recorded, copy the final image set from the output work folder to a deliverable image folder, then assemble the image-based PPTX:

```powershell
New-Item -ItemType Directory -Force -Path "outputs/slides/<project>/<version>-images" | Out-Null
Copy-Item "outputs/slides/<project>/<version>-work/redesigned/slide*_redesigned.jpg" `
  "outputs/slides/<project>/<version>-images/" `
  -Force
```

```powershell
python ".agents/skills/udds-codex-slide-generator/scripts/assemble_pptx.py" `
  --review outputs/slides/<project>/<version>-work/archetype_review.md `
  --img outputs/slides/<project>/<version>-images `
  --output outputs/slides/<project>/<version>.pptx
```

Assembly uses the slide image files in `outputs/slides/<project>/<version>-images` as the final packaging source of truth, not `archetype_review.md`. It sorts `slide*_redesigned.jpg/png` by slide number and optional letter suffix, so insert slides like `slide21A_redesigned.jpg`, `slide31B_redesigned.jpg`, or `slide42C_redesigned.jpg` appear immediately after `slide21_redesigned.jpg`, `slide31B_redesigned.jpg`, or `slide42C_redesigned.jpg`.

You may pass `--review outputs/slides/<project>/<version>-work/archetype_review.md` for console labels only; it does not control packaging order or omit extra insert slides.

By default, assembly archives per-slide `slide*_redesigned.*.record.json` files into the image folder's `records/` after the PPTX is created, so packaging folders are not cluttered by recorder metadata. Use `--record-json-mode keep` for audit-heavy runs or `--record-json-mode delete` only when the user explicitly wants those metadata files removed.

### Stage 6. Final Gemini Image QA And Decision Report

QA is the final decision/reporting step. For typo QA, it audits only Vietnamese spelling, diacritics, and OCR/rendering noise in the already assembled slide image set. It must not audit content completeness, factual correctness, brand quality, layout, or aesthetics unless the visual issue directly makes text unreadable. It must not delete, omit, move aside, overwrite, regenerate, or fix slides by itself.

Prepare Gemini image QA jobs:

```powershell
python ".agents/skills/udds-codex-slide-generator/scripts/prepare_qa_jobs.py" `
  --input outputs/slides/<project>/<version>-work/redesigned `
  --manifest outputs/slides/<project>/<version>-work/deck_manifest.json
```

Run Gemini image QA with the API key from `.env`:

```powershell
python ".agents/skills/udds-codex-slide-generator/scripts/run_gemini_image_qa.py" `
  --jobs outputs/slides/<project>/<version>-work/redesigned/gemini_qa_jobs.json `
  --env .env
```

Then:

1. Read `gemini_qa_prompt.md`.
2. For every slide in `gemini_qa_jobs.json`, Gemini reads the slide image and uses `expected_text` only as a spelling/diacritic reference.
3. Inspect only Vietnamese typo/OCR classes: wrong or noisy tone marks; a/ă/â, o/ô/ơ, u/ư, e/ê, d/đ mistakes; merged stacked accents; broken letters; unreadable pseudo-text inside Vietnamese words.
4. Write `gemini_qa_report.json` and `gemini_qa_typo_flags.md` as findings reports only.

Human-in-the-loop QA gate:

- Treat Gemini visual QA/OCR as fallible, especially for Vietnamese diacritics, dense slide text, and sketchnote handwriting.
- Gemini QA is allowed to mark suspected typo/OCR/diacritic issues and recommend `Targeted Typo Fix` or `Human Review`, but it is not approval to act.
- Do not flag extra or missing content merely because `expected_text` is incomplete. If visible text is valid Vietnamese and readable, it passes typo QA even when the content differs from the manifest.
- Do not run a targeted fix, aesthetic tweak, regenerate, delete, overwrite, or move-aside operation from QA findings alone.
- First present a concise slide-by-slide findings list to the user, including exact observed text, expected text, confidence, and recommended action.
- Wait for explicit user approval naming the slide(s) and accepted action(s).
- Only after approval, prepare the approved targeted fix or regeneration job.
- Never delete or omit slides after QA. Keep the full slide set intact and only mark slides needing review.
- If using `gemini_qa_report.json` to drive scripts, encode human approval in `qa_approved_actions.json`; otherwise `prepare_image_tool_jobs.py` must refuse QA-driven tweak/regenerate jobs.

If the user approves an `Aesthetic Tweak: ...`, run `prepare_image_tool_jobs.py` again only for the approved slide/action.

If the user approves `Regenerate: ...`, move aside the failed slide image only after that approval, update the manifest if text density caused the issue, then run `prepare_image_tool_jobs.py` for the approved slide/action.

Then repeat Stage 4 only for the approved pending job.

## Notes

- Do not overlay editable text on top of generated slide images unless the user explicitly asks for a hybrid deck.
